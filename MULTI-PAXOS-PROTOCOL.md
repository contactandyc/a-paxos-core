# Multi-Paxos Protocol Specification

## 0. Scope and failure model

This specification describes a crash-recovery Multi-Paxos protocol for a replicated deterministic state machine.

It assumes:

```text
Failure model: crash-stop and crash-recovery only
No Byzantine behavior
Messages may be lost, duplicated, delayed, or reordered
Stable storage is correct
Nodes may restart after crashes
Safety does not depend on timing
Liveness requires eventual synchrony and eventually one stable leader
```

A value is **chosen** for a slot when a quorum of acceptors in the slot’s governing configuration has accepted the same value for that slot.

A value is **committed** when some node knows that the value is chosen.

A value is **applied** when a learner executes the committed command against the state machine.

The protocol supports:

```text
Fixed-cluster Multi-Paxos
Leader election and Phase-1 recovery
Pipelined Phase-2 replication
In-order state-machine application
Linearizable writes
Linearizable reads
Crash recovery
Learner catch-up
Snapshots and log compaction
Safe dynamic reconfiguration
Client request deduplication
```

---

# 1. Core model

The cluster maintains a replicated log:

```text
slot 1 -> command A
slot 2 -> command B
slot 3 -> no-op
slot 4 -> command C
...
```

All non-faulty replicas eventually learn the same chosen value for every committed slot and apply those values in ascending slot order.

Each physical server usually acts as:

```text
Proposer  -> tries to get values chosen
Acceptor  -> stores promises and accepted values
Learner   -> learns chosen values and applies them
Replica   -> hosts the state machine and client session table
```

A **leader** is the currently active distinguished proposer.

Multi-Paxos optimization:

```text
Basic Paxos:
    Phase 1 + Phase 2 per slot

Multi-Paxos:
    Phase 1 once per leadership ballot
    Phase 2 directly for many slots
```

# 2. Identifiers

## 2.1 Ballots

A ballot is a totally ordered identifier:

```text
ballot = (epoch_number, node_id)
```

Ordering is lexicographic:

```text
(epoch_a, node_a) > (epoch_b, node_b)
```

iff:

```text
epoch_a > epoch_b
```

or:

```text
epoch_a == epoch_b and node_a > node_b
```

Each node MUST persist enough information to guarantee that it never reuses an old ballot after crash/restart.

A node choosing a new ballot MUST choose one greater than:

```text
its persisted max_generated_ballot
its persisted promised_ballot
all persisted accepted_ballot values
all ballots observed in NACKs
```

## 2.2 Slots

Slots are positive monotonically increasing log indexes:

```text
1, 2, 3, ...
```

Each Paxos slot is logically independent for safety, but learners apply slots in order.

## 2.3 Values

A log value is one of:

```text
ClientCommand(client_id, sequence_number, operation)
NoOp
ReadBarrier(read_id)
BeginJointConfig(old_config, new_config)
FinalizeConfig(new_config)
```

# 3. Quorums and configurations

For a normal configuration:

```text
Config C = set of voting acceptor IDs
Quorum(C) = any majority of C
```

A value for slot `s` governed by configuration `C` is chosen when accepted by a majority of `C`.

For joint reconfiguration:

```text
JointConfig J(old, new)
Quorum(J) = majority(old) AND majority(new)
```

A value governed by a joint configuration is chosen only when accepted by:

```text
one majority of old_config
and
one majority of new_config
```

This guarantees quorum intersection across:

```text
old -> joint
joint -> new
```

# 4. Persistent state

Every voting node MUST persist the following before replying to relevant RPCs.

## 4.1 Acceptor persistent state

```text
promised_ballot:
    Highest ballot this acceptor has promised not to go below.

accepted:
    Map slot -> AcceptedEntry

AcceptedEntry:
    slot
    accepted_ballot
    accepted_value
    config_epoch_id
```

Invariant:

```text
An acceptor MUST NOT accept two different values for the same (slot, ballot).
```

It MAY replace an older accepted value for a slot with a value from a higher ballot, but only by processing a valid `Accept` message.

## 4.2 Ballot persistent state

```text
max_generated_ballot:
    Highest ballot ever generated by this node.
```

Before sending any `Prepare`, a node MUST persist the ballot it generated.

## 4.3 Learner persistent state

```text
chosen_log:
    Map slot -> chosen value

commit_index:
    Highest contiguous known-chosen slot

apply_index:
    Highest slot applied to the state machine
```

`commit_index` and `apply_index` may be reconstructed from durable log and snapshot metadata, but production implementations SHOULD persist them for fast restart.

## 4.4 Snapshot persistent state

```text
snapshot_index:
    Highest slot included in the snapshot

snapshot_state:
    Serialized deterministic state machine state

snapshot_session_table:
    Client deduplication table

snapshot_config_state:
    Active or joint configuration at snapshot_index

snapshot_digest:
    Optional integrity hash
```

A node MUST NOT discard accepted or chosen log entries for slot `i` unless:

```text
i <= snapshot_index
and
slot i is known chosen
and
slot i has been applied
and
the snapshot is durably stored
```

# 5. Volatile state

## 5.1 Leader volatile state

```text
leader_state:
    inactive | recovering | active

active_ballot:
    Current leadership ballot

next_slot:
    Next slot available for new client proposals

pending_accepts:
    slot -> set of acceptor IDs that accepted the value

pending_values:
    slot -> value

chosen:
    slot -> value known chosen by this leader

client_waiters:
    slot -> client request waiting for response
```

## 5.2 Replica volatile state

```text
known_leader:
    Best known current leader

last_observed_ballot:
    Highest ballot observed in any message

catchup_state:
    Tracks missing log ranges and snapshot transfers
```

# 6. RPC schema

## 6.1 Prepare

```text
Prepare {
    ballot
    leader_id
    config_epoch_id
    first_uncommitted_slot
}
```

`first_uncommitted_slot` is usually:

```text
local commit_index + 1
```

It tells acceptors where to begin returning accepted suffix entries.

## 6.2 Promise

```text
Promise {
    acceptor_id
    ballot
    promised_ballot
    snapshot_index
    commit_index
    accepted_suffix: [
        {
            slot
            accepted_ballot
            accepted_value
            config_epoch_id
        }
    ]
}
```

The suffix includes all accepted entries with:

```text
slot >= max(first_uncommitted_slot, snapshot_index + 1)
```

## 6.3 Accept

```text
Accept {
    ballot
    leader_id
    config_epoch_id
    slot
    value
    leader_commit_index
}
```

`leader_commit_index` acts as a commit notification piggyback.

## 6.4 Accepted

```text
Accepted {
    acceptor_id
    ballot
    config_epoch_id
    slot
    value_hash
    acceptor_commit_index
}
```

## 6.5 NACK

```text
NACK {
    acceptor_id
    promised_ballot
    optional_known_leader
}
```

A node receiving a NACK with a ballot greater than its own active ballot MUST step down.

## 6.6 Heartbeat / CommitNotice

```text
CommitNotice {
    leader_id
    ballot
    config_epoch_id
    commit_index
}
```

## 6.7 Catch-up RPCs

```text
FetchEntries {
    requester_id
    start_slot
    end_slot
}

FetchEntriesResponse {
    entries: [
        {
            slot
            chosen_value
            config_epoch_id
        }
    ]
    snapshot_required: bool
    snapshot_index
}

InstallSnapshot {
    snapshot_index
    snapshot_state
    snapshot_session_table
    snapshot_config_state
    snapshot_digest
}
```

# 7. Acceptor rules

## 7.1 Handling Prepare

On receiving:

```text
Prepare(b, leader_id, config_epoch_id, first_uncommitted_slot)
```

the acceptor executes:

```text
if b < promised_ballot:
    reply NACK(promised_ballot)

else if b == promised_ballot:
    reply Promise again idempotently

else if b > promised_ballot:
    promised_ballot = b
    persist promised_ballot
    reply Promise
```

The Promise MUST include all accepted suffix entries at or above:

```text
max(first_uncommitted_slot, snapshot_index + 1)
```

A repeated `Prepare` for the same ballot is not an error.

## 7.2 Handling Accept

On receiving:

```text
Accept(b, slot, value)
```

the acceptor executes:

```text
if slot <= snapshot_index:
    reject and return snapshot_index

if b < promised_ballot:
    reply NACK(promised_ballot)

if accepted[slot].accepted_ballot == b
   and accepted[slot].accepted_value != value:
       reject as protocol violation

promised_ballot = max(promised_ballot, b)
accepted[slot] = (slot, b, value, config_epoch_id)

persist promised_ballot and accepted[slot]

reply Accepted(b, slot, hash(value))
```

The acceptor MUST persist before replying `Accepted`.

Duplicate identical `Accept` messages are idempotent.

Conflicting same-ballot `Accept` messages are fatal protocol violations.

# 8. Leader election and activation

## 8.1 Becoming a candidate

A node may campaign when it suspects the leader has failed.

It chooses a fresh ballot:

```text
b = next_ballot_greater_than_all_observed_or_persisted_ballots()
persist max_generated_ballot = b
leader_state = recovering
```

It sends:

```text
Prepare(b, self_id, current_config_epoch, commit_index + 1)
```

to all voting acceptors in the current governing configuration.

## 8.2 Promise quorum

The candidate waits for a quorum of `Promise` replies.

For a normal config:

```text
majority(config)
```

For a joint config:

```text
majority(old_config) AND majority(new_config)
```

If it receives a higher-ballot NACK, it updates `last_observed_ballot` and steps down.

## 8.3 Phase-1 recovery merge

Let `Q` be the quorum of Promise responses.

Let:

```text
start = local commit_index + 1
S = union of all accepted_suffix entries returned by Q
max_seen_slot = highest slot present in S
```

For every slot:

```text
i in [start, max_seen_slot]
```

the candidate applies this rule:

```text
if any Promise returned accepted value(s) for slot i:
    choose the value with the highest accepted_ballot for slot i

else:
    choose NoOp
```

If two different values appear with the same highest accepted ballot for the same slot, the implementation MUST stop; this indicates a broken proposer or corrupted storage.

## 8.4 Recovery re-proposal

The candidate re-proposes every recovered value under its new ballot:

```text
Accept(active_ballot, slot i, recovered_value)
```

It waits until each recovered slot has a current-ballot quorum certificate.

Only after all recovered slots through `max_seen_slot` are chosen may the node become active:

```text
commit_index = max_seen_slot, if contiguous
next_slot = max_seen_slot + 1
leader_state = active
```

This conservative rule is simpler than allowing new client proposals during recovery.

## 8.5 Leader no-op

After recovery, the leader SHOULD commit a `NoOp` or `ReadBarrier` in its own ballot before serving optimized reads.

This proves the leader has successfully completed Phase 2 in its current ballot.

# 9. Normal write path

## 9.1 Client submits command

A client sends:

```text
ClientCommand(client_id, sequence_number, operation)
```

to any node.

If the node is not leader, it either:

```text
redirects the client to known_leader
```

or:

```text
forwards the request to the leader
```

## 9.2 Leader proposes

The active leader assigns:

```text
slot = next_slot
next_slot += 1
```

Then sends:

```text
Accept(active_ballot, slot, command, leader_commit_index)
```

to all acceptors in the governing configuration.

## 9.3 Accept quorum

When the leader receives a valid quorum of:

```text
Accepted(active_ballot, slot, hash(command))
```

the value is chosen.

The leader records:

```text
chosen_log[slot] = command
```

Then advances:

```text
commit_index
```

to the highest contiguous chosen slot.

## 9.4 Client response

The leader SHOULD return a client response only after:

```text
the command is chosen
and
all previous slots are applied
and
this command is applied
and
the state-machine response is known
```

This gives linearizable write semantics.

# 10. Learning and application

## 10.1 Commit propagation

The leader advertises `commit_index` by:

```text
piggybacking it on Accept messages
sending periodic CommitNotice heartbeats
responding to FetchEntries requests
```

Learners receiving a commit notice do not blindly apply through that index. They first ensure they have every chosen value up to that index.

```text
on CommitNotice(commit_index = K):

    if missing any slot <= K:
        FetchEntries(missing_range)

    once slots 1..K are locally known:
        local commit_index = max(local commit_index, K)
        apply in order
```

## 10.2 In-order application

Learners apply only contiguous committed entries:

```text
while chosen_log[apply_index + 1] exists:
    apply_index += 1
    apply chosen_log[apply_index]
```

A learner MUST NOT apply slot `s` before applying all slots `< s`.

`NoOp` entries do nothing except advance the log.

# 11. Client semantics and deduplication

The state machine maintains a durable session table:

```text
session_table[client_id] = {
    last_sequence_number
    last_response
}
```

This spec assumes each client has at most one outstanding request at a time.

On applying a client command:

```text
if sequence_number <= session_table[client_id].last_sequence_number:
    return cached last_response

else:
    execute operation
    store response
    update last_sequence_number
    update last_response
```

The session table MUST be included in snapshots.

This provides exactly-once execution with respect to client retries, assuming stable `client_id` and monotonically increasing per-client sequence numbers.

# 12. Reads

The protocol supports two linearizable read modes.

## 12.1 Read through the log

This is the default and simplest correct read path.

The leader proposes:

```text
ReadBarrier(read_id)
```

or a read command as a normal log entry.

Once the read entry is chosen and applied, the leader evaluates the read against the local state machine and returns the result.

This is always linearizable but costs a normal Paxos write round.

## 12.2 Optimized quorum read

The leader may serve a read without logging it only if all conditions hold:

```text
leader_state == active
Phase-1 recovery has completed
leader has committed at least one entry in active_ballot
no reconfiguration is currently pending
```

Read algorithm:

```text
read_index = commit_index

send ReadProbe(active_ballot) to acceptors

wait for a quorum of ReadProbeOK replies confirming:
    acceptor has not promised a higher ballot

if any higher-ballot NACK is received:
    step down and retry through new leader

wait until apply_index >= read_index

serve read from local state machine
```

This is linearizable because the quorum probe confirms that no higher-ballot leader had already obtained a conflicting leadership quorum before the read barrier point.

Lease reads are not part of the core protocol. They require explicit bounded-clock-skew assumptions.

# 13. Gaps and out-of-order slots

Multi-Paxos allows acceptors to accept slots out of order.

Example:

```text
slot 10 accepted
slot 11 accepted
slot 13 accepted
slot 12 missing
```

The leader may know slot 13 is chosen, but it cannot advance `commit_index` past 11 until slot 12 is chosen.

Gap handling:

```text
if a slot below a chosen suffix remains unresolved:
    leader proposes NoOp for that slot

if a new leader discovers no accepted value for a gap during Phase 1:
    it proposes NoOp for that slot

if any accepted value exists for the slot:
    it must recover the highest accepted-ballot value
```

A NoOp is safe only when Phase 1 proves that no accepted value for that slot exists in the recovery quorum.

# 14. Crash recovery

## 14.1 Restarting node

On restart, a node:

```text
loads promised_ballot
loads max_generated_ballot
loads accepted entries
loads snapshot
loads chosen log
reconstructs commit_index and apply_index
starts as non-leader
```

A node MUST NOT resume as leader after restart using volatile leadership state.

It must run Phase 1 again with a fresh higher ballot.

## 14.2 Lost disk

If a node loses its persistent acceptor state, it MUST NOT rejoin using the same identity.

Safe options:

```text
rebuild it as a new non-voting learner
catch it up from snapshot and log
add it back through reconfiguration
```

or:

```text
operator restores a valid durable backup
```

A wiped acceptor reusing the same identity can violate Paxos safety.

## 14.3 Leader crash after Phase 1

If a leader completes Phase 1 but crashes before Phase 2, no value is chosen solely by Phase 1.

A future leader uses a higher ballot, runs Phase 1, and either:

```text
recovers accepted values
```

or:

```text
fills empty slots with NoOp
```

## 14.4 Leader crash after partial Phase 2

If a leader sends Accept messages for a slot but crashes before learning a quorum, a later leader’s Phase 1 will discover any accepted values and re-propose the highest accepted-ballot value.

# 15. Snapshots and compaction

## 15.1 Snapshot creation

A node may snapshot after applying a committed prefix:

```text
snapshot_index = K
```

The snapshot MUST include:

```text
state machine state at K
client session table at K
configuration state at K
last included slot K
optional digest/checksum
```

The node persists the snapshot atomically.

## 15.2 Log truncation

After durable snapshot creation, the node may discard:

```text
chosen_log[slot] for slot <= K
accepted[slot] for slot <= K
```

but must retain:

```text
promised_ballot
max_generated_ballot
snapshot metadata
configuration state
```

## 15.3 Phase-1 interaction with snapshots

If an acceptor receives:

```text
Prepare(b, first_uncommitted_slot)
```

where:

```text
first_uncommitted_slot <= snapshot_index
```

it replies with:

```text
snapshot_index
accepted_suffix starting at snapshot_index + 1
```

The candidate must install or otherwise account for the snapshot before becoming active.

## 15.4 Catch-up with snapshots

If a lagging node requests entries older than the leader’s retained log:

```text
FetchEntries(start_slot <= snapshot_index)
```

the leader replies:

```text
snapshot_required = true
snapshot_index = leader.snapshot_index
```

The lagging node installs the snapshot, sets:

```text
apply_index = snapshot_index
commit_index = snapshot_index
```

and then fetches log entries after the snapshot.

# 16. Dynamic reconfiguration

This spec uses a conservative two-phase joint-configuration protocol. It does not use the `i + alpha` pipeline scheme as the normative protocol.

Only one reconfiguration may be pending at a time.

## 16.1 Configuration states

The cluster is always in one of:

```text
Normal(C)
Joint(old_config, new_config)
```

## 16.2 Governance rule

A slot is governed by the configuration active when that slot is proposed.

More specifically:

```text
BeginJoint slot:
    governed by old_config

Slots after BeginJoint and before Finalize:
    governed by Joint(old_config, new_config)

Finalize slot:
    governed by Joint(old_config, new_config)

Slots after Finalize:
    governed by new_config
```

## 16.3 Adding nodes

Before a new node may vote, it must be bootstrapped as a non-voting learner.

It must receive:

```text
latest snapshot
committed log suffix after snapshot
current configuration state
```

Only then may it participate in the joint configuration.

## 16.4 Begin joint configuration

Current leader in `Normal(C_old)` proposes:

```text
BeginJointConfig(C_old, C_new)
```

as a normal log value under `C_old`.

It is chosen by:

```text
majority(C_old)
```

After this entry is committed and applied, the configuration becomes:

```text
Joint(C_old, C_new)
```

The leader stops normal client writes while reconfiguration is pending.

## 16.5 Leadership in joint configuration

After entering joint configuration, the leader MUST establish authority under the joint quorum.

It chooses a fresh higher ballot and runs Phase 1 against:

```text
majority(C_old) AND majority(C_new)
```

It recovers any suffix values according to the normal Phase-1 recovery rule.

Only after this succeeds may it propose the finalize entry.

## 16.6 Finalize new configuration

The leader proposes:

```text
FinalizeConfig(C_new)
```

under the joint configuration.

It is chosen only after receiving Accepted responses from:

```text
majority(C_old) AND majority(C_new)
```

After `FinalizeConfig(C_new)` is committed and applied, the active configuration becomes:

```text
Normal(C_new)
```

## 16.7 Leadership in new configuration

After finalization, the leader SHOULD run Phase 1 under `C_new` with a fresh higher ballot before resuming client writes.

This fences removed nodes and establishes clean leadership in the new configuration.

## 16.8 Removed nodes

Nodes removed from the configuration:

```text
stop voting after learning FinalizeConfig
may remain as non-voting learners
must not be counted in future quorums
```

If a removed node has stale state and accepts obsolete messages, its responses are ignored because it is no longer a voter in the governing configuration.

## 16.9 Failure during reconfiguration

If the leader crashes after `BeginJointConfig` is chosen but before `FinalizeConfig` is chosen:

```text
future leaders must recover the log
observe the committed BeginJointConfig
enter Joint(C_old, C_new)
run Phase 1 using joint quorums
continue or abort by committing an explicit config command
```

A leader MUST NOT skip over an uncommitted or unapplied configuration entry.

A leader MUST NOT propose user commands past a known reconfiguration barrier until the governing configuration for those slots is known.

# 17. Safety invariants

A correct implementation preserves these invariants.

## 17.1 Ballot uniqueness

```text
No two proposers generate the same ballot.
No proposer reuses a ballot after crash.
```

## 17.2 Promise monotonicity

```text
promised_ballot never decreases.
```

## 17.3 Accepted-value persistence

```text
An acceptor persists accepted values before replying Accepted.
```

## 17.4 Same-ballot consistency

```text
A proposer never sends two different values for the same slot and ballot.
An acceptor never accepts two different values for the same slot and ballot.
```

## 17.5 Phase-1 value selection

For every recovered slot:

```text
if any acceptor in the Phase-1 quorum reports an accepted value,
the leader must choose the value with the highest accepted ballot.
```

## 17.6 Quorum intersection

For any slot, every Phase-1 quorum intersects every possible Phase-2 quorum that could have chosen a value for that slot.

For reconfiguration, this is enforced by joint quorums.

## 17.7 In-order application

```text
A learner never applies slot s before all slots < s are applied.
```

## 17.8 Snapshot safety

```text
A slot may be compacted only after it is chosen, applied, and included in a durable snapshot.
```

# 18. Liveness rules

Paxos safety holds without timing assumptions, but progress requires:

```text
eventual message delivery among a quorum
eventual stable leader
bounded leader-election disruption
durable storage availability
```

Implementations SHOULD use:

```text
randomized election timeouts
exponential backoff after NACK
leader heartbeats
failure-detector timeouts
batching
pipelining
periodic catch-up
```

On receiving a higher-ballot NACK, a leader MUST step down.

---

# 19. Implementation checklist

A complete implementation should verify:

```text
[ ] Ballots are globally unique and never reused.
[ ] promised_ballot is persisted before Promise.
[ ] accepted[slot] is persisted before Accepted.
[ ] accepted values are stored per slot, not globally.
[ ] same (slot, ballot) cannot accept different values.
[ ] Phase-1 recovery merges per slot.
[ ] gaps are filled with NoOp only when safe.
[ ] leader does not serve clients until recovery completes.
[ ] commit_index advances only over contiguous chosen slots.
[ ] learners fetch missing entries before applying.
[ ] client commands carry client_id and sequence_number.
[ ] session table is snapshotted.
[ ] reads are either logged or protected by quorum-read rules.
[ ] snapshots include state, session table, and config state.
[ ] compacted slots are known chosen and applied.
[ ] wiped acceptors never rejoin under the same identity.
[ ] reconfiguration uses joint quorums or another proven quorum-overlap protocol.
[ ] removed nodes are not counted in new quorums.
[ ] leader steps down on higher-ballot NACK.
```
